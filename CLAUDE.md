# CLAUDE.md

本文件是本项目的永久开发规范。以后每次开始开发前，都必须先阅读本文件，以及 `BUSINESS_RULES.md`、`ARCHITECTURE.md`、`CHANGELOG.md`。

## 1. 项目定位

本项目是长期维护、长期运营的软件，不是 Demo，不是练习项目，不是一次性项目。

任何修改都必须以现有功能稳定为前提，避免引入回归。

## 2. 修改原则

任何需求都必须先分析、先定位、先确认原因，再修改。

禁止猜测，禁止边改边试，默认采用最小修改原则。

除非用户明确要求，不允许：

- 大面积重构
- 修改无关代码
- 修改正常工作的功能
- 修改无关页面

## 3. 数据安全原则

任何修改不得造成：

- 数据丢失
- 数据覆盖
- 重复数据
- 历史数据变化
- 金额变化
- 房源编号变化
- 房间编号变化
- 租客资料变化

数据库修改必须兼容旧数据。

## 4. UI 原则

默认保持颜色、布局、按钮位置、字体、交互不变。

只有用户明确提出时，才允许修改 UI。

## 5. Mobile First

所有开发优先保证 iPhone 和 Android 使用体验，桌面端只是兼容。

## 6. 修复流程

必须严格遵循以下流程：

1. 阅读需求
2. 阅读相关代码
3. 分析原因
4. 确定修改文件
5. 开始修改
6. 完成后自测
7. 确认没有影响其它模块
8. 提交修改结果

## 7. 输出规范

完成修改后必须说明：

- 修改了哪些文件
- 为什么修改
- 是否影响其它模块
- 是否需要继续测试

## 8. 用户需求最高优先级

如果本规范与用户最新要求冲突，始终以用户最新要求为最高优先级。

## 9. 上下文读取原则

为了提高效率、降低 Token 消耗，默认遵循：

- 每次开发先阅读 `CLAUDE.md`
- 再阅读 `BUSINESS_RULES.md`
- 再阅读 `ARCHITECTURE.md`
- 再阅读 `CHANGELOG.md`

然后只读取本次需求涉及的模块代码。

默认不要扫描整个项目，不要重新分析整个仓库，不要重新读取所有文件。

只有确定存在跨模块影响时，才允许扩大阅读范围。

始终遵循：

**先文档，后代码。**

**先局部，后整体。**

**Minimum Context.**

## 10. 业务规则优先原则

如果 `BUSINESS_RULES.md` 已存在对应业务规则：

- 不要重新设计业务逻辑
- 不要根据自己的理解修改业务规则
- 默认严格按照 `BUSINESS_RULES.md` 实现
- 只有用户明确要求修改业务规则时，才允许更新 `BUSINESS_RULES.md`

`BUSINESS_RULES.md` 永远作为项目业务规则唯一可信来源（Single Source of Truth）。

## 11. 新开发任务启动规则

当用户提出一项新的开发需求，而不是继续上一项尚未完成的任务时，开始分析前必须按以下顺序阅读：

1. `CLAUDE.md`
2. `BUSINESS_RULES.md`
3. `ARCHITECTURE.md`
4. `CHANGELOG.md`

阅读完成后，依次理解项目开发规范、当前业务规则、项目架构和最近修改记录，再分析本次需求。

## 12. 最小读取与最小影响原则

- 默认只读取本次需求直接涉及的模块代码。
- 修改收款时，优先只读取收款相关代码；修改租客时，优先只读取租客相关代码。
- 不默认扫描整个项目，不默认重新分析全部代码，始终保持 Minimum Context。
- 如果确认本次修改会影响其它模块，必须先向用户说明影响范围和原因，等待用户确认后再扩大修改范围。
- 除非用户明确要求，不得修改数据库结构、API、已有字段、已有 UI 或其它正常工作的功能。
- 每次修改完成后，必须在 `CHANGELOG.md` 末尾追加记录，不得覆盖历史记录。

## 13. 业务规则唯一来源

`BUSINESS_RULES.md` 是本项目业务规则的唯一可信来源。若其中已有对应规则，开发时必须严格遵守，不得自行推测、重新设计或用个人理解替换现有规则。只有用户明确要求变更业务规则时，才允许同步更新 `BUSINESS_RULES.md`。
