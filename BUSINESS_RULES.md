# BUSINESS_RULES.md

本文件记录本项目的永久业务规则。以后每确定一条新的业务规则，都应追加到这里。

## 1. 房源与房间

- 房源、房间、租客、合同、收款、支出、押金等数据必须保留历史。
- 页面展示可以变化，但底层业务数据不能随意丢失或覆盖。
- 房间当前状态应根据当前有效租住关系动态计算，不能只靠手动状态判断。

## 2. 租客与退租

- 退租表示当前租住关系结束。
- 退租后不再催租，不再计算欠费，不再进入欠费提醒。
- 退租、归档、作废不应删除历史收款、押金、合同附件、历史流水。
- 退租后如果没有其他有效租客覆盖，房间应自动变为空置。

## 3. 欠费规则

- 只有当前在租租客才参与欠费计算。
- 已退租、已归档、已结束、非在租的租客不参与欠费统计。
- 欠费以租金覆盖结束日期和当前有效租金流水为准。

## 4. 收款规则

- 收款流水是唯一真实来源。
- 收款记录必须保留完整历史。
- 收款统计、首页统计、利润统计、提醒逻辑都应从收款流水重新计算。
- 收款记录删除后，相关租客、房间、首页、提醒应同步重新计算。

## 5. 月租、实收与覆盖期

- 月租标准仅用于参考和显示。
- 本次实收金额用于实际收入、利润和合伙结算。
- 租金覆盖开始日期与结束日期由用户手动维护。
- 收款日期用于归属财务月份统计。
- 财务月份、首页收入、利润、结算、按月筛选均严格按收款日期归属；租金覆盖期不得改变收入归属月份。
- 同一笔收款只能生成一条财务流水并统计一次；房租和押金可在同一流水中保存，但不得因覆盖跨月重复计入收入。
- 收款管理列表主行显示该笔 `amount_paid` 的本次合计收入；房租金额、押金金额和其他收入明细只在展开详情中拆分显示。

## 6. 押金规则

- 押金状态只作为管理提醒，不自动生成支出。
- 押金状态不参与利润计算。
- 真正退押金时，必须由用户手动登记支出。
- 押金收入与押金退还应通过收支流水体现。

## 7. 合伙结算

- 收款归属和付款归属必须可追踪。
- 合伙结算必须根据收款流水和支出流水计算。
- A/B 只是固定默认选项，也允许自定义归属名称。

## 8. 附件规则

- 合同附件、收款附件、支出票据等业务文件必须存 Supabase Storage，不得存入 GitHub 仓库。
- 删除附件只删除对应 Storage 文件和数据库记录，不影响代码仓库。

## 9. 权限规则

- 普通用户可查看、编辑、登记、退租、归档。
- 永久删除仅允许管理员执行。
- 永久删除必须二次确认。

## 10. 默认行为

- 没有明确业务规则时，优先采用最小改动和兼容旧数据的方式实现。
- 所有新增规则必须兼容已有历史数据和已有操作流程。

- 租客列表的金额只读取租客当前月租标准字段，不得从当月收款、最近一次实收、押金金额、首页统计或租金覆盖期临时计算。
- 续交房租时，当前月租标准与本次实收房租相互独立；仅当用户主动修改当前月租标准时，才更新租客资料中的月租标准。本次实收不同不会自动改写月租标准。
- 租客当前月租标准缺失或为 0 时，兼容修复只能读取该租客按收款日期最新的一笔有效房租金额，不能读取押金或房租加押金合计；已有非零标准不得被历史流水覆盖。
- 租客列表主行优先显示房间名称或房间编号；房源长名称只作辅助信息，不能挤掉房间编号。
## 11. 月租标准数据来源与租客列表
- 租客当前月租标准唯一保存于 tenants.monthly_rent，租客列表和租客详情只读取该字段。
- 收款记录中的 rent_payments.amount_due 表示本次房租金额；amount_paid 表示本次实收合计，不能用押金或合计金额覆盖月租标准。
- 收款登记自动创建租客时，如果本次房租金额大于 0，必须把该房租金额写入 tenants.monthly_rent。
- 对历史数据兼容修复时，只允许把 monthly_rent 为空或 0 的租客补为按收款日期最新的一笔有效房租金额；已有非零月租标准不得被历史流水覆盖。
- 租客列表主行优先显示房间名称或房间编号；房源完整名称只在详情中显示，不能挤掉房间号。
- 租金覆盖到期提醒统一依据当前在租租客的最新租金覆盖结束日期计算：剩余31天及以上不提醒，30至16天为黄色“剩余X天”，15至1天为橙色“即将到期X天”，0天为红色“今日到期”，过期为红色“已到期X天”。
- 已退租、已归档、已结束或非在租租客不参与租金到期提醒、欠费计算和提醒中心。
- 租客管理首次进入默认按提醒紧急程度排序：红色、橙色、黄色、正常；同等级按租金覆盖结束日期升序。手动到期日、月租、房源、状态排序只改变当前筛选结果的显示顺序，不修改数据库。

## 12. 账号与权限

- 系统账号类型只允许 `owner` 和 `custom`，只读、收款录入等均是 `custom` 的权限配置，不新增角色类型。
- 固定主管理员为邮箱 `主管理员保密邮箱（仅数据库与安全配置）`、Auth User ID `主管理员保密 Auth ID（仅数据库与安全配置）`；正式 owner 身份以数据库资料为准，不依赖前端环境变量。
- 固定主管理员必须保持启用、拥有全部房源、全部模块、全部敏感权限和永久删除权限，不允许被停用、降级或删除。
- 自定义账号默认不授予模块、房源、敏感权限或写权限；只有主管理员主动授权后才可使用。
- 房源授权必须使用 `property_id`，并覆盖房间、租客、收支、押金、提醒、统计、利润、结算、附件、搜索、导出和历史流水。
- 权限必须同时在页面、服务端和数据库 RLS 生效；仅隐藏按钮不构成权限控制。
- `SUPABASE_SERVICE_ROLE_KEY` 只允许服务器端主管理操作使用，不得进入浏览器、日志、Git 或普通业务查询。
- 账号停用后，即使旧 Access Token 尚未自然过期，数据库权限也必须依据 `user_profiles.status` 立即拒绝访问。
- `app_sessions` 只记录 Supabase JWT 的 `session_id` 和撤销状态，不保存 Refresh Token；无法可靠区分设备前，第一版只实现强制退出账号全部设备。
- 操作日志和安全日志必须追加写入，普通账号和业务删除权限均不能修改或删除日志；真实操作人必须由服务端根据已验证 Token 确定。
- 阶段一只建立权限基础结构、固定 owner 和兼容 RLS，不改变现有登录页面和业务数据访问流程；自定义账号及全业务权限接入必须等待阶段验收后继续。

### 阶段二账号管理规则

- 账号列表、新建、权限保存、房源授权、密码重置、停用、启用和强制退出仅允许固定 owner 执行；custom 即使拥有保存的管理账号预留字段，也不能在阶段二管理账号。
- 自定义账号登录名全局唯一且不区分大小写；登录名映射到服务端私有的内部 Auth 邮箱，浏览器、账号列表和日志均不返回内部邮箱。
- 自定义账号默认无房源、无模块权限、无敏感权限；全部房源使用 property_access_mode=all，指定房源只在 user_property_access 保存真实 property_id。
- 自定义账号停用后必须立即被服务端和 RLS 拒绝；启用后原权限保留，但必须重新登录。
- 强制退出当前只支持该账号全部设备：撤销该账号所有 app_sessions，custom 账号的精确 session_id 随即失效。Supabase 无管理员 API 可按 user ID 立即撤销既有 Access Token，因此 Refresh Token 的最终撤销仍由 Supabase 会话刷新生命周期控制；账号停用则无论旧 Token 是否到期都由 status 立即拒绝。
- 密码只保存在 Supabase Auth 的哈希中。重置密码、停用、启用、强制退出、创建账号和权限或房源范围变更都必须由服务端根据已验证 Token 写入追加式安全日志。
- 阶段二只接入账号管理页、账号管理服务端接口、应用会话和日志基础。全部业务页面的模块按钮隐藏、字段脱敏、按房源数据过滤和统计隔离属于阶段三。

### 阶段三全业务权限规则

- 所有菜单和页面入口按模块查看权限显示；利润、合伙结算和操作日志还必须同时具备对应敏感权限。未授权入口隐藏，直接访问网址时显示无权限页面。
- 新增、编辑、归档、永久删除及附件查看、下载、上传、替换、删除按钮按模块操作权限和敏感权限共同显示；隐藏按钮不能替代服务端和 RLS 校验。
- 普通业务写入必须携带当前 Supabase Access Token，通过统一服务端业务写接口校验账号状态、应用会话、模块操作和 `property_id`，随后仍以当前登录用户身份执行并接受 RLS 校验；不得使用 Service Role 绕过 RLS。
- 自定义账号的业务行 `user_id` 始终保存主管理员 workspace owner ID；账号读取的是主管理员数据空间，不建立独立业务副本。
- 全部房源模式读取主管理员全部房源；指定房源模式只允许读取 `user_property_access` 中保存的真实 `property_id` 及其关联业务数据。列表、详情、统计、利润、提醒、结算和附件均继承同一范围。
- 租客电话、微信和备注由数据库脱敏 RPC 返回；缺少敏感权限时，接口不得返回完整值。证件字段在当前业务表不存在时不新增或伪造字段。
- 操作日志默认追加记录业务表新增、修改、归档和删除，以及附件查看、下载和元数据变化；日志查询需要操作日志模块查看权限和敏感日志查看权限。账号管理仍只允许 owner。
- 汇总页面只读取账号已获查看权限的数据模块，单个未授权子模块不得导致整页加载失败；完整经营汇总账号应同时授予所需基础业务模块的查看权限。
- 永久删除必须拥有对应模块删除权限并继续遵守原有二次确认和关联数据保护规则。

## 账号分享与自助密码（2026-07-15）

- 只有 owner 可以分享 custom 账号的登录信息；分享内容固定为系统名称、正式登录地址和自定义登录账号，不得包含密码、内部认证邮箱、Auth User ID、令牌或免密链接。
- 分享或复制登录信息必须写入追加式安全日志。停用账号可以被标记和查看，但不作为默认分享目标。
- 所有状态为 active 的 owner 和 custom 账号均可在个人中心修改自己的密码。必须验证当前密码，新密码至少 8 位、与确认值一致且不能与当前密码相同。
- 自助修改密码仅能修改当前真实登录账号；管理员重置他人密码仍使用 owner 专用流程。
- 密码仅由 Supabase Auth 保存。成功修改后撤销该账号应用会话并要求重新登录；日志只能记录成功或失败，不得记录密码、密码哈希、令牌或 Cookie。

## 登录信息隐私（2026-07-15）

- 公开登录页面、前端帮助文字、接口响应、前端日志和可见维护文档不得展示主管理员真实邮箱、Auth User ID、内部认证邮箱、令牌或密码。
- 主管理员仍可通过其真实邮箱正常登录；该邮箱仅可保留在数据库迁移和服务器端安全校验中。

## 长期登录与应用恢复（2026-07-18）

- 长期登录依赖 Supabase 持久化 Refresh Token 和短期 Access Token 自动刷新；不得人为延长 Access Token，也不得用前端计时器模拟长期登录。
- 首次恢复本地 Session 时允许显示一次认证初始化；站内导航、Safari/PWA 回到前台、`SIGNED_IN` 重发和 `TOKEN_REFRESHED` 只能静默校验，不得清空账号、权限、房源范围或业务数据快照。
- 正常网络错误或恢复竞态不得等同于退出、停用或无权限。只有明确退出、账号停用、精确 `session_id` 被撤销或 Refresh Token 确认失效时才清除登录状态。
- Supabase Session 有效且账号 active 时，可以补建意外缺失的 `app_sessions` 行；明确 revoked 的 `session_id` 或早于全设备撤销边界的会话不得恢复。
- 普通业务读写必须先取得当前有效短期 Token；并发恢复只允许一次 Refresh Token 轮换，避免页面校验和保存请求分别使用新旧 Token。
