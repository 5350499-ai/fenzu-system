# BUSINESS_RULES.md

本文件记录本项目的永久业务规则。以后每确定一条新的业务规则，都应追加到这里。

## 1. 房源与房间

- 房源、房间、租客、合同、收款、支出、押金等数据必须保留历史。
- 页面展示可以变化，但底层业务数据不能随意丢失或覆盖。
- 房间当前状态应根据当前有效租住关系动态计算，不能只靠手动状态判断。

## 2. 租客与退租

- 退租表示当前租住关系结束。
- 退租后不再催租，不再计算欠费，不再进入欠费提醒。
- 退租、归档、作废不应删除历史收款、押金、合同附件、历史流水。
- 退租后如果没有其他有效租客覆盖，房间应自动变为空置。

## 3. 欠费规则

- 只有当前在租租客才参与欠费计算。
- 已退租、已归档、已结束、非在租的租客不参与欠费统计。
- 欠费以租金覆盖结束日期和当前有效租金流水为准。

## 4. 收款规则

- 收款流水是唯一真实来源。
- 收款记录必须保留完整历史。
- 收款统计、首页统计、利润统计、提醒逻辑都应从收款流水重新计算。
- 收款记录删除后，相关租客、房间、首页、提醒应同步重新计算。

## 5. 月租、实收与覆盖期

- 月租标准仅用于参考和显示。
- 本次实收金额用于实际收入、利润和合伙结算。
- 租金覆盖开始日期与结束日期由用户手动维护。
- 收款日期用于归属财务月份统计。
- 财务月份、首页收入、利润、结算、按月筛选均严格按收款日期归属；租金覆盖期不得改变收入归属月份。
- 同一笔收款只能生成一条财务流水并统计一次；房租和押金可在同一流水中保存，但不得因覆盖跨月重复计入收入。

## 6. 押金规则

- 押金状态只作为管理提醒，不自动生成支出。
- 押金状态不参与利润计算。
- 真正退押金时，必须由用户手动登记支出。
- 押金收入与押金退还应通过收支流水体现。

## 7. 合伙结算

- 收款归属和付款归属必须可追踪。
- 合伙结算必须根据收款流水和支出流水计算。
- A/B 只是固定默认选项，也允许自定义归属名称。

## 8. 附件规则

- 合同附件、收款附件、支出票据等业务文件必须存 Supabase Storage，不得存入 GitHub 仓库。
- 删除附件只删除对应 Storage 文件和数据库记录，不影响代码仓库。

## 9. 权限规则

- 普通用户可查看、编辑、登记、退租、归档。
- 永久删除仅允许管理员执行。
- 永久删除必须二次确认。

## 10. 默认行为

- 没有明确业务规则时，优先采用最小改动和兼容旧数据的方式实现。
- 所有新增规则必须兼容已有历史数据和已有操作流程。

- 租客列表的金额只读取租客当前月租标准字段，不得从当月收款、最近一次实收、押金金额、首页统计或租金覆盖期临时计算。
- 续交房租时，当前月租标准与本次实收房租相互独立；仅当用户主动修改当前月租标准时，才更新租客资料中的月租标准。本次实收不同不会自动改写月租标准。
- 租客当前月租标准缺失或为 0 时，兼容修复只能读取该租客按收款日期最新的一笔有效房租金额，不能读取押金或房租加押金合计；已有非零标准不得被历史流水覆盖。
- 租客列表主行优先显示房间名称或房间编号；房源长名称只作辅助信息，不能挤掉房间编号。
## 11. 月租标准数据来源与租客列表
- 租客当前月租标准唯一保存于 tenants.monthly_rent，租客列表和租客详情只读取该字段。
- 收款记录中的 rent_payments.amount_due 表示本次房租金额；amount_paid 表示本次实收合计，不能用押金或合计金额覆盖月租标准。
- 收款登记自动创建租客时，如果本次房租金额大于 0，必须把该房租金额写入 tenants.monthly_rent。
- 对历史数据兼容修复时，只允许把 monthly_rent 为空或 0 的租客补为按收款日期最新的一笔有效房租金额；已有非零月租标准不得被历史流水覆盖。
- 租客列表主行优先显示房间名称或房间编号；房源完整名称只在详情中显示，不能挤掉房间号。
- 租金覆盖到期提醒统一依据当前在租租客的最新租金覆盖结束日期计算：剩余31天及以上不提醒，30至16天为黄色“剩余X天”，15至1天为橙色“即将到期X天”，0天为红色“今日到期”，过期为红色“已到期X天”。
- 已退租、已归档、已结束或非在租租客不参与租金到期提醒、欠费计算和提醒中心。
- 租客管理首次进入默认按提醒紧急程度排序：红色、橙色、黄色、正常；同等级按租金覆盖结束日期升序。手动到期日、月租、房源、状态排序只改变当前筛选结果的显示顺序，不修改数据库。
